## 🌟 Overview

**FitZone** is a production-ready, AI-enhanced fitness management system designed for gyms, fitness studios, and personal trainers. It combines traditional gym management (members, classes, payments, attendance) with cutting-edge AI and ML features including churn prediction, AI-generated workout plans, real-time nutrition analysis, QR-based check-in, and revenue forecasting.

Built entirely in **Python/Flask**, it runs on SQLite for development and MySQL for production, requires zero cloud deployment to get started, and ships with a full demo data seed so you can see everything working in under 60 seconds.

> 💡 **No API key required** — all AI features have smart mock fallbacks so the app runs fully offline. Add a `GEMINI_API_KEY` to unlock real generative AI.

### Step 2 — Create a virtual environment
```bash
python -m venv .venv

# Windows
.venv\Scripts\activate

source .venv/bin/activate
```

### Step 3 — Install dependencies
```bash
pip install -r requirements.txt
```

### Step 4 — Set environment variables (optional)
```bash
# .env file
GEMINI_API_KEY=your_google_gemini_api_key_here
SECRET_KEY=your_custom_secret_key
DB_TYPE=sqlite        # or 'mysql' for production
```

### Step 5 — Run the application
```bash
python app.py
```

The database is created and seeded with demo data automatically on first run.

## 🤖 AI & ML Modules

### `ai_services.py` — Google Gemini Integration

All AI functions follow a **Gemini-first, mock-fallback** pattern. If `GEMINI_API_KEY` is set and valid, real LLM responses are used. Otherwise, smart rule-based mock responses maintain full functionality.

#### `generate_workout_plan(member_data, duration_weeks)`
Generates a structured weekly training plan based on member's goals, height, weight, and medical conditions.

```python
result = generate_workout_plan(
    member_data={'fitness_goal': 'Muscle Gain', 'weight': 78, 'height': 175, 'gender': 'male'},
    duration_weeks=4
)
# Returns: {'title': str, 'plan_content': str, 'ai_model': str}
```

#### `get_chatbot_response(user_message, user_context)`
Context-aware fitness assistant with multi-turn conversation support.

```python
response = get_chatbot_response(
    user_message="How many calories should I eat?",
    user_context={'first_name': 'Ali', 'fitness_goal': 'Weight Loss'}
)
# Returns: str (assistant reply)
```

#### `generate_payment_reminder(member_info, invoice_info, tone)`
Writes a payment reminder in one of three tones: `'friendly'`, `'professional'`, or `'firm'`.

```python
message = generate_payment_reminder(
    member_info={'first_name': 'Ali', 'last_name': 'Khan'},
    invoice_info={'invoice_number': 'INV-2026-0001', 'total_amount': 89.99, 'days_overdue': 5},
    tone='friendly'
)
# Returns: str (reminder message)
```

#### `analyze_nutrition(food_description, member_data)`
Parses a plain-text meal description into calorie and macro estimates.

```python
result = analyze_nutrition(
    food_description="2 scrambled eggs, toast, and a banana",
    member_data={'fitness_goal': 'Muscle Gain', 'weight': 80}
)
# Returns: {'calories': int, 'protein_g': float, 'carbs_g': float,
#            'fat_g': float, 'fiber_g': float, 'analysis': str, 'ai_model': str}
```

### `ml_services.py` — Machine Learning

#### 1. Churn Predictor

Uses **Logistic Regression** (scikit-learn) trained on live attendance and payment data.

**Features used:**
| Feature | Signal |
|---|---|
| `days_since_last_visit` | Recency — most important signal |
| `visits_last_30` | Short-term frequency |
| `visits_last_90` | Long-term frequency |
| `avg_duration_minutes` | Engagement depth |
| `overdue_invoices` | Payment health |
| `days_to_membership_end` | Renewal risk |
| `has_workout_plan` | AI engagement (reduces churn) |
| `progress_records_count` | Measurement engagement |

**Risk labels:**
- 🔴 `high` — score ≥ 0.65
- 🟡 `medium` — score 0.35–0.64
- 🟢 `low` — score < 0.35

```python
from ml_services import predict_churn_all_members, get_churn_summary

results = predict_churn_all_members()
# Returns: [{'member_id': int, 'score': float, 'label': str}, ...]

summary = get_churn_summary()
# Returns: {'high': int, 'medium': int, 'low': int, 'unscored': int}
```

> ⚠️ **Fallback**: When fewer than 5 members exist or scikit-learn is unavailable, a hand-crafted rule-based scoring function is used automatically.

#### 2. Revenue Forecast

Uses **Linear Regression** on 12 months of actual payment data to project the next 3 months.

```python
from ml_services import forecast_revenue

data = forecast_revenue(months_ahead=3)
# Returns: [{'month': 'Jan 2026', 'revenue': 1250.0, 'is_forecast': False}, ...]
# Last 3 entries have is_forecast=True
```

#### 3. Peak Hour Analysis

Analyses attendance `check_in` timestamps to find the busiest hours of the day.

```python
from ml_services import get_peak_hours

hours = get_peak_hours()
# Returns: [{'hour': 6, 'label': '06:00', 'avg_visits': 3.2, 'total_visits': 96}, ...]
```

## 👥 User Roles

FitZone has three roles. Access is controlled by the `@role_required()` decorator on every route.

| Role | Access Level |
|---|---|
| 🔴 **admin** | Full access — all routes, all members, all data, ML tools |
| 🟡 **trainer** | Members list, attendance, classes, progress records |
| 🟢 **member** | Own profile, own progress, workout plans, nutrition log, QR code, chatbot |

### Role-based navigation
The sidebar automatically adjusts to show only the routes the logged-in user can access. Admin-only sections (Churn Predictor, Revenue Forecast, Analytics, All QR Codes, Payment Reminders) are hidden for trainers and members.

## 🔒 Security

FitZone follows Flask security best practices. Below are the measures implemented and recommendations for production deployment.

### ✅ Implemented

#### Authentication & Sessions
- **Bcrypt password hashing** — all passwords are hashed with `flask-bcrypt` (bcrypt rounds = 12 by default). Plaintext passwords are never stored.
- **Session management** — Flask-Login manages sessions with `remember_me` support and 7-day cookie expiry.
- **CSRF protection** — Flask-WTF provides CSRF tokens on all state-changing forms.
- **Secure cookie flags** — `SESSION_COOKIE_HTTPONLY=True` and `SESSION_COOKIE_SAMESITE='Lax'` are set. Enable `SESSION_COOKIE_SECURE=True` in production behind HTTPS.

#### Access Control
- **Role-based access control (RBAC)** — every admin/trainer-only route is protected with `@role_required(['admin'])` or `@role_required(['admin', 'trainer'])`.
- **Login required** — every route (except `/login` and `/register`) requires authentication via `@login_required`.
- **Resource ownership** — member-facing routes validate that the requesting user owns the resource (e.g., a member cannot view another member's nutrition logs or QR token).

#### QR Check-In
- **Cryptographically secure tokens** — QR tokens are generated with `secrets.token_urlsafe(32)` (256 bits of randomness).
- **Token invalidation** — regenerating a QR code deactivates all previous tokens instantly.
- **Duplicate check-in prevention** — the validation logic checks for an existing open attendance record before creating a new one.

#### API Keys
- **Environment variable only** — `GEMINI_API_KEY` is read from the OS environment or `.env` file and never committed to source code.
- **No key in templates** — the API key is never passed to Jinja2 templates or returned in API responses.

#### Database
- **SQLAlchemy ORM** — all database queries use parameterized ORM calls, preventing SQL injection.
- **Soft deletes** — members and trainers are deactivated (`is_active=False`) rather than hard-deleted, preserving referential integrity.
- **Pool health checks** — `pool_pre_ping=True` prevents stale connections.

### ⚠️ Before Going to Production

> These steps are **required** before deploying publicly.

#### 1. Change the secret key
```env
# .env
SECRET_KEY=generate-a-long-random-string-here
```
Generate one with:
```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

#### 2. Enable HTTPS and secure cookies
```python
# config.py — ProductionConfig
SESSION_COOKIE_SECURE = True   # Only send cookie over HTTPS
```

#### 3. Switch to MySQL
```env
DB_TYPE=mysql
DB_HOST=your-db-host
DB_USER=fitzone_user
DB_PASSWORD=strong-password
DB_NAME=fitzone_prod
```

#### 4. Set DEBUG=False
```python
# config.py — ProductionConfig
DEBUG = False
```
Or use the production config:
```python
app.config.from_object(config['production'])
```

#### 5. Rate limiting (recommended)
Install and configure `Flask-Limiter` on the `/api/chatbot` and `/qr/validate` endpoints to prevent abuse:
```bash
pip install Flask-Limiter
```

#### 6. Never commit secrets
Add to `.gitignore`:
```
.env
instance/
.venv/
__pycache__/
*.pyc
```

#### 7. Security headers
Consider adding `flask-talisman` for automatic HTTPS redirect, HSTS, and Content Security Policy headers:
```bash
pip install flask-talisman
```

## 🤝 Contributing

Contributions are welcome! FitZone is an open project — whether you fix a bug, add a feature, improve documentation, or suggest an idea.

### How to contribute

#### 1. Fork and clone
```bash
git clone https://github.com/yourname/fitzone.git
cd fitzone
git checkout -b feature/your-feature-name
```

#### 2. Set up development environment
```bash
python -m venv .venv
source .venv/bin/activate        # or .venv\Scripts\activate on Windows
pip install -r requirements.txt
python app.py
```

#### 3. Make your changes

Follow these conventions:
- **Code style** — PEP 8 compliant. Use meaningful variable names.
- **Route structure** — all routes go in `app.py`. New AI features belong in `ai_services.py`. New ML logic belongs in `ml_services.py`.
- **New models** — add to `models.py` and include the relationship on the parent model.
- **Templates** — extend `base.html`. Use Bootstrap 5 utility classes. No inline `<style>` blocks.
- **Comments** — add docstrings to all new functions.

#### 4. Test your changes
```bash
# Run the smoke test suite
python test_ai.py

# Verify the app starts cleanly with fresh DB
rm -f instance/fitness.db
python app.py
```

Manually verify:
- [ ] Login works for admin, trainer, and member roles
- [ ] Your new feature is accessible from the correct role
- [ ] No existing routes are broken
- [ ] The churn model still runs without errors

#### 5. Open a pull request

- Use a clear, descriptive title: `feat: add wearable sync endpoint` or `fix: churn score not updating on membership expiry`
- Describe what you changed and why
- Reference any related issue: `Closes #42`

### 🐛 Reporting Bugs

Open a GitHub issue with:
1. Steps to reproduce
2. Expected vs actual behaviour
3. Python version and OS
4. Relevant error output or traceback

### 💡 Suggesting Features

Open a GitHub issue with the `enhancement` label. Describe:
1. The problem you're trying to solve
2. Your proposed solution
3. Any alternative approaches you considered
